# ルール v2 提案 — メカニクス借用による「ジレンマ強化」たたき台

> **このファイルは提案(discussion draft)であり、ルールの正ではない。**
> 章ごとに採否・数値を議論し、確定した章から [RULES.md](RULES.md) に反映 → GameConfig フラグ付きで実装する。
>
> 背景:ステージ1のプレイテストで「どこに置くかのジレンマが薄い」という課題が出た。
> 原因は ①トークンが余る ②誰がどこに置いても同じ(代替可能) ③個人目標がチーム行動と衝突しない
> ④全公開・同時相談ゆえの奉行問題(1人が全員の最適手を指示できる) ⑤ドラマではなく計算になっている
> (正解が見えると作業化する)、の5点。
> 名作ボードゲームのメカニクスを借用して、これらを構造的に解決する。

## 導入ロードマップ(複雑性の予算管理)

一度に全部入れず、1段階ずつプレイテストして採否を決める。

## §1 炎上システム 🔥(借用元:パンデミック)

**狙い**:放置の代償を作る。「全部やればいい」を壊し、消火(守り)と建設(攻め)で
トークンの使い道を競合させる。

### ルール案

1. **炎上デッキ**を新設。各タスクタイルを指す対象カードで構成する
2. フェーズ開始時、既存のイベントカードに加えて**炎上カードを `firePerPhase`(暫定2)枚**引き、
   対象タスクに🔥トークンを1個ずつ置く(解決済みタスクが対象なら何も起きない)
3. 🔥1個につき、そのタスクの**必要行動トークン +1**(スコープクリープ。要件が膨らんでいく)
4. **延焼(アウトブレイク)**:同一タスクに **`fireOutbreakThreshold`(暫定4)個目**の🔥は置けず、
   代わりに依存ツリーの**子タスク全員に🔥1個ずつ** + **CS -1**。子がいなければ同列の各タスクへ
5. **消火**:プランニング中、行動トークン1個で自タスク・自系統の🔥1個を除去(`extinguishCost: 1`)。
   資格外のタスク(§2)は2個必要
6. **大炎上カード**(エピデミック相当)を炎上デッキに `epidemicCount`(暫定2)枚混ぜる:
   引いたら炎上捨て札をリシャッフルして山に戻し、任意のタスク1枚に🔥2個
   (= 一度燃えたタスクはまた燃えやすい)

### 既存ルールへの影響

- §2-1(フェーズ開始)に「炎上カードを引く」を追加
- §2-3(実行)の手順1は「必要トークン数 = 基本値 + 🔥数」で判定
- 未解決タスクの持ち越し時、🔥も持ち越す(燃え続ける)

### GameConfig 追加(暫定値)

```
fireEnabled: true
firePerPhase: 2
fireOutbreakThreshold: 4
fireOutbreakCsPenalty: 1
extinguishCost: 1          // 資格内
extinguishCostOutside: 2   // 資格外(§2導入後)
epidemicCount: 2
```

### 未決事項

- ❓ 解決時に🔥が残っていた場合の追加ペナルティ(疲労+1?)は必要か → 暫定:なし(必要トークン増のみ)
- ❓ 消火は実行ステップ中にもできるべきか → 暫定:プランニング中のみ
- ❓ 大炎上の対象選択は誰が決めるか → 暫定:PM

## §2 配属トリアージ(改訂版。借用元:ダーウィンズジャーニー資格ゲート + パンデミック)

> **本章は旧「§2 資格ワーカー」を置き換える本命案。** 旧案(ワーカー個体管理)は複雑性が高いため、
> 末尾に「代替案(重量版)」として要約のみ残す。
>
> ステージ1の課題①〜⑤(冒頭)の本丸。**配置に痛みを作る**ことを、3つの軸で実現する:
> - **軸①** スキルを個人に紐づける(代替不能性 → 課題②④)
> - **軸②** 供給 < 需要(配属トリアージ → 課題①⑤)
> - **軸③** メモリ付きの脅威で計画を揺らす(予測できるのに手が回らない → 課題⑤)
>
> 学習(スキル獲得)は**主役にしない**。毎フェーズの習得ループを中心に置くと「自分のスキル番付を
> 育てる個人パズル」に化け、プロジェクト体験が痩せる。学習は「工数の機会費用を払う痛い投資」に留める。

### 軸① スキルはプレイヤー本人に紐づく(ワーカー個体は導入しない)

1. v1 の「プレイヤーのスキルレベル」を中核に格上げ。各プレイヤーは3系統
   (ディレクション/デザイン/エンジニアリング)の **スキルプロファイル** を持つ。
   ロールで初期値が非対称(例:ディレクター=ディレクションLv2、デザイナー=デザインLv2、他系統Lv0〜1)
2. 行動トークンは **そのプレイヤーの工数**。タスクに置いたトークンは置いた本人のスキルを持ち込む。
   **= 誰が置くかで結果が変わる**(課題②「代替可能」の破壊)
3. タスクの「スキル条件」を **席ゲート** に変更:
   - **専門席**(タイルに0〜2席):系統×必要Lv を指定。その系統 Lv 以上のプレイヤーの工数でのみ充足
   - **人手席**:無条件。誰の工数でもよい(積み上げ式)
   - 専門席が埋まらないとタスクは通常解決できない(→ 軸②の三択へ)
   - **過剰スペック割引**:Lv2 を Lv1 専門席に充てると実行コスト -`overqualifiedDiscount`(暫定1)
4. ワーカー個体管理は採らない(複雑性の予算を軸②③に回す)。スキルはあくまで **プレイヤー単位**。

### 軸② 供給 < 需要 → 毎フェーズのトリアージ

設計の要:**専門席の総需要 > チームの専門家工数** になるよう数値を組む(`demandSeatRatio` で管理)。
全部に適任は当てられない。埋まらない専門席ごとに、3択を突きつける:

1. **やっつけ(無資格充当)**:Lv 不足の工数で強行。代償(三重は重すぎ注意・要間引き):
   - 成果物が1段ダウン(Lv2→Lv1、Lv1→0)
   - 実行時疲労 +`understaffFatigue`(暫定1)
   - 解決後も🔥1個が残る(`understaffLeavesFire` → 軸③と接続:雑な仕事は燃える)
2. **育成**:工数を学習に回し、来フェーズの適任を作る(今フェーズは手薄になる。後述)
3. **外注(追加請求)**:`outsourceBudgetCost`(暫定:予算4)+ **CS -`outsourceCsCost`**(暫定1)で
   専門席1つを自動充足。金で殴れるが CS が痛い(`outsourcePerPhase` でフェーズ回数制限)

`mismatchEnabled: false` で「満たすか満たさないか」の v1 単純スキル条件に戻せる(リグレッション維持)。

### 学習 = 痛い投資(毎ラウンドの儀式にしない)

- 学習タイルにトークンを置く = **その工数は今フェーズ消える**。来フェーズ開始時に該当系統 +1 Lv
- 毎フェーズ確定でスキルが伸びる設計にはしない。「育成に回す余裕がない」局面を作るのが目的
  (= 習得ループの主役化を防ぎ、プロジェクト体験を守る。ユーザー懸念への直接の答え)
- 昇格コスト:`trainTokens = 現Lv + 1` 個の工数(高Lvほど重い)

### 軸③ メモリ付きの脅威で配属計画を揺らす

イベント/炎上は **QCD直撃より「需要側(配属パズル)」を揺らす** 方向に寄せる。
脅威は「理不尽な被弾」ではなく **「来るのが半分わかるのに手が回らない」**(パンデミックの感染デッキ=メモリ)に:

1. **要件フリップ**(秘匿要件の動的版・`requirementFlipEnabled`):実行直前に専門席の要求系統が変わる
   (デザイン席→エンジニアリング席)。「固めたのに違った」
2. **スコープ増(炎上)**:🔥は既存どおり必要トークン +1。適任が手一杯のタスクに偏らせる。
   炎上デッキのリシャッフル(§1/§10)で半予測可能を担保
3. **稼働離脱**:限界イベント等で「特定プレイヤーが次フェーズ工数 -N」= 突然のスキル不足。
   狙いを **個人/特定タスク** に絞り、全体一律にしない(所有権を守る=課題④の悪化防止)
4. **予告マーク**(`threatTelegraph`):脅威は次フェーズの発火を盤上に予告できる(可視の蓄積)

### 既存ルールへの影響

- §2-2(プランニング):配置の単位に「誰の工数か」の意味が加わる
- §2-3 手順2(スキル条件)→ 専門席ゲート + やっつけ三択 に置換
- §4(疲労):やっつけ疲労 `understaffFatigue` を追加
- §5(スキルと成長):本章で全面改訂(スキル=プレイヤー紐付け、学習=工数の機会費用)
- §3 §4(本提案):本章の上に乗る。資源クラフト(§3)は専門席と、産出エンジン(§4)は稼働タスクと接続

### GameConfig 追加(暫定値)

```
assignmentTriageEnabled: true
skillSeatsEnabled: true            // 専門席ゲート
overqualifiedDiscount: 1           // Lv2をLv1席に充てた時のコスト割引
mismatchEnabled: true              // やっつけ充当を許可
understaffFatigue: 1               // やっつけ時の追加疲労
understaffLeavesFire: true         // やっつけ解決後に🔥1個残す
outsourceBudgetCost: 4             // 外注で専門席1つを充足する予算
outsourceCsCost: 1
outsourcePerPhase: 1               // 外注の回数上限/フェーズ
trainTokensFormula: "currentLv+1"  // 昇格に要する工数
demandSeatRatio: 1.3               // 専門席総需要 / 専門家工数(>1=供給不足。要シム調整)
requirementFlipEnabled: true       // 要件フリップ・イベント
threatTelegraph: true              // 脅威の次フェーズ予告マーク
```

### 未決事項

- ❓ `demandSeatRatio` の最適値(きつすぎると詰む・緩いとトリアージが消える)→ ボットシミュで計測
- ❓ やっつけの代償「Lvダウン+疲労+🔥」は三重で重すぎないか → 1〜2個に間引く前提でテスト
- ❓ 専門席を複数人の Lv 合算で満たせるか(Lv1×2人 = Lv2席)→ 暫定:不可(合算は人手席のみ)
- ❓ 外注が予算で殴る抜け道になりすぎないか → `outsourcePerPhase` で上限。要バランス
- ❓ ロール別の初期スキルプロファイル具体値(RULES.md §5・§9-6 の TODO と統合)

### 代替案(重量版・不採用):資格ワーカー個体管理

行動トークンをワーカー個体に置き換え、各ワーカーに資格マークを持たせて「人を育てる」方式(旧 §2)。
代替不能性はより強いが、個体管理の煩雑さ(識別・育成中のロック・物理版のミープル+リング)が
複雑性予算を圧迫する。スキルをプレイヤー単位に置く本命案で物足りなければ再検討する。

## §3 資源と高品質化(借用元:パンデミックの特効薬)

**狙い**:Lv2 成果物を「重いタスクのおまけ」から「チームで狙って作る中期プロジェクト」へ。
品質戦略がエンジン構築(§4)と接続する。

### ルール案

1. **資源3種**を新設:📋知見(ディレクション)/ 🎨意匠(デザイン)/ ⚙技術(エンジニアリング)。
   チーム共有ではなく**個人の手元**に入る(EP・個人目標と絡むため)
2. 入手手段:
   - タスク解決時の産出(タイル定義を「Lv2成果物」→「資源1〜2個」に差し替え。獲得は参加者で分配、
     端数は PM が割り振る)
   - §4 導入後:稼働タスクの毎フェーズ産出
3. **高品質化(クラフト)**:プランニング中、行動トークン(ワーカー)1個 +
   **同系統の資源 `craftCost`(暫定3)個**を支払うと、その系統の **Lv2 成果物**を獲得する
   (= ウィルス特効薬。5枚同色カードの感覚)
4. Lv1 成果物は従来どおりタスク解決で獲得。**タイル直付けのV2 成果物は廃止**し、本ルートに一本化
5. 品質(Q)判定は変更なし:そのフェーズに獲得した Lv2 成果物数で判定
   → 「今フェーズ中にクラフトが間に合うか」の計画ゲームになる

### 既存ルールへの影響

- §6 タスクタイル定義:`deliverables` の Lv2 を資源産出フィールドに置き換え
- 限界イベント「品質の妥協」:Lv2成果物の劣化 → **資源2個を失う**に変更(劣化対象がない事故を防ぐ)
- 要件カード「ポートフォリオ掲載OK」:ボーナス成果物 → 資源+1 に変更

### GameConfig 追加(暫定値)

```
resourceCraftEnabled: true
craftCost: 3            // 同系統資源でLv2成果物1個
craftActionCost: 1      // 消費する行動トークン(ワーカー)数
mixedCraftCost: 4       // ❓ 異系統混合で作れるなら(暫定:不可)
```

### 未決事項

- ❓ 資源の持ち越し上限(インフレ防止)→ 暫定:上限なしで一度テスト
- ❓ 資源の譲渡(パンデミックの「知識の共有」のような同タスク上での受け渡し)→ 暫定:あり。
  同じタスクに参加したプレイヤー間でのみ譲渡可(これも相談と配置の理由になる)

## §4 タスクドラフトと産出エンジン(借用元:イッツアワンダフルワールド)

**狙い**:「どのタスクを場に出すと良いエンジンになるか」をチームの議論にする。
スコープを自分たちで決める = 納期判定の母数を自分たちで握る。

### ルール案

1. タスクタイルの自動公開を廃止し、**タスクカードの手札**にする:
   フェーズ開始時、各プレイヤーに `taskCardsPerPlayer`(暫定3)枚配る
2. **スコープ会議**(新ステップ。フェーズ開始とプランニングの間):
   全員で相談し、手札から**合計 `tasksPerPhase` 枚まで**を場(依存レーン)に出す
   - カードには**列番号**(起点 / 中盤 / 仕上げ)が刷ってあり、その列にしか置けない
   - 一部のカードは**前列条件**を持つ(例:「中盤列に『設計』タグのタスクがあること」)
3. **スコープカット**:場に出さなかったカードは捨てて、1枚につき**予算+1 または 資源1個**
   (IWW のリサイクル。「断る勇気」に報酬を)
4. **産出(エンジン)**:一部のタスクは解決後に**稼働状態**になり、
   以後毎フェーズ開始時に資源や予算を産出する(例:スタイルガイド → 毎フェーズ🎨1)
   - 稼働タスクは盤面に残り続ける(🔥の延焼対象にもなる = 動いてるシステムは燃える)
5. **納期(D)判定**:母数は「場に出したタスク」のみ。出しすぎは自殺、絞りすぎは品質・産出が痩せる

### 既存ルールへの影響

- §2-1(フェーズ開始)を「配布 → 炎上 → イベント → スコープ会議 → 補充」に再構成
- §7(セットアップ)4 のタスク公開 → 初期手札配布 + 第1スコープ会議に置き換え
- タスクカードに新フィールド:列番号 / 前列条件 / タグ / 産出(稼働時)

### GameConfig 追加(暫定値)

```
taskDraftEnabled: true
taskCardsPerPlayer: 3
scopeCutBudget: 1        // カット1枚あたりの予算回復
maxBoardTasks: 6         // 場に出せる上限
```

### 未決事項

- ❓ 手札は個人持ちか共有か → 暫定:**個人持ち・中身は公開**(相談はできるが、出す/カットの最終決定は持ち主。
  奉行対策と交渉の種)
- ❓ 使わなかった手札の持ち越し → 暫定:持ち越し可、上限5枚
- ❓ タグとプロジェクトカードのシナジー(TM式)はこの章に含めるか → 暫定:含める(カード定義時に設計)

## §5 EP とマイルストーン(借用元:白ブラス + テラフォーミングマーズ)

**狙い**:奉行問題への二段構え。「利他が利己になる」(ブラス)と「公開の早取り競争」(TM)で、
全員が自分の意思で動く理由を作る。

### ルール案

#### EP(Engagement Point)の定義 — RULES.md §9-9 の答え

> **EP = 自分の仕事が他人に使われた回数。**

EP が入る契機(各 +1):

1. 自分が作成者の**成果物**が、他プレイヤーの高品質化・タスク解決条件に使われた
2. 自分が**親タスク**に参加していて、その子タスクを他プレイヤー(自分不参加)が解決した
   (= 自分の下準備の上で他人が成果を出した)
3. 自分の**稼働タスク**(§4)の産出資源を他プレイヤーが使った
4. 自分が**消火**(§1)したタスクを、そのフェーズ中に他プレイヤーが解決した

- EP は公開トラック(隠すと集計が辛い。物理版を考慮)
- 個人目標カードに EP 条件を追加:「EP 5 以上」「EP でチーム1位」など

#### マイルストーン(公開・早取り)

セットアップ時に3枚公開。**最初に達成した1人だけ**が取れる:

| 例 | 条件 |
|---|---|
| 火消し番長 | 累計消火 5 回 |
| 品質職人 | Lv2 成果物の作成 2 個 |
| 育成の人 | 資格付与/昇格 3 回 |
| 縁の下 | EP 4 |
| 鬼の工数 | 1フェーズにワーカー5体以上をタスクに配置 |

#### 個人勝利条件の再定義

> 個人勝利 = **チーム勝利** かつ(**秘匿の個人目標達成** または **マイルストーン獲得**)

- 秘匿目標は配られた2枚から1枚選ぶ方式に変更(自分の戦略に合わせられる)
- チーム敗北時は従来どおり全員敗北(ここは協力ゲームの背骨なので変えない)

### GameConfig 追加(暫定値)

```
epEnabled: true
milestonesEnabled: true
milestoneCount: 3
personalGoalChoices: 2   // 配られる枚数(1枚選ぶ)
```

### 未決事項

- ❓ EP のインフレ調整(契機が多すぎないか)→ プレイテストで契機を間引く前提
- ❓ マイルストーンの同時達成 → 暫定:アクション解決順で先の者
