# スマートプロジェクト — ルール仕様書

> **このファイルがルールの単一の正(Single Source of Truth)です。**
> 元になったスプレッドシート(基本設計/リソース)と食い違う場合、本ファイルを優先します。
> シート間の矛盾は「リソース」シートを正として解消済みです。
>
> 各項目には確度ラベルを付けています:
> - ✅ **確定** — 実装してよい
> - 🔧 **暫定** — 初期値として実装するが、`GameConfig` で変更可能にすること
> - ❓ **TODO** — 未決定。実装前に決めるか、決め打ちせずパラメータ化すること

## 1. ゲーム概要 ✅

- 4〜5人協力型ボードゲーム(デジタル版を先行開発)
- テーマ:Web制作会社のクライアントプロジェクト
- プレイヤーは異なるロール(PM/ディレクター/デザイナー/エンジニア)を担当し、QCD(品質・コスト・納期)のバランスを取りながらプロジェクトを成功させる
- チーム共通の勝利条件に加え、各自が秘匿の個人目標を持つ(準協力型)

### 勝敗条件 ✅

| 条件 | 内容 |
|---|---|
| チーム勝利 | 最終フェーズ終了時に CSトラック ≥ 0 |
| チーム敗北 | CSトラックが 0 未満になった時点(❓ 即時敗北か最終判定のみかは TODO → 暫定:**即時敗北**) |
| 個人勝利 | チーム勝利 かつ(個人目標カードの条件達成 **または** マイルストーン獲得 🔧 v2.1 §11) |

チームが敗北した場合、個人目標の達成は評価されない。

### ❓ TODO:5人プレイ時のロール構成
ロールは4種。5人目はロール重複か、第5ロール(例:マーケター/QAエンジニア)新設かを決める。
暫定実装:**4人プレイのみ対応**で開始し、エンジンはプレイヤー数可変で設計しておく。

## 2. ゲームの流れ ✅

1ゲーム = **4フェーズ**:

1. 企画・要件定義
2. 設計・デザイン
3. 開発
4. テスト

各フェーズは以下の4ステップで進行する。

### 2-1. フェーズ開始

- タスクタイルを規定枚数(🔧 5〜6枚)公開し、依存関係に従ってタスクエリアに配置する
- 炎上カードを `firePerPhase` 枚引き、対象タスクに🔥トークンを置く 🔧(v2.1 §10。`fireEnabled: false` で無効化可)
- イベントカードを1枚引いて解決する ✅(毎フェーズ開始時 + タスクのイベントマーク解決時に発動)
- 各プレイヤーは行動トークンを補充する(🔧 基本値:5個/フェーズ。疲労Lv2のプレイヤーは -1)

### 2-2. プランニング(同時配置) ✅

- **手番なし。全員が相談しながら同時に**行動トークンをタスクタイルに配置する
- 他プレイヤーが着手したタスクへの**応援配置(積み上げ)が可能**。タスクの必要トークン数は複数人・複数フェーズにまたがって積み上げてよい
- 配置済みトークンはフェーズ終了まで盤上に残る
- 休憩スペース・追加請求スペースへの配置もこのステップで行う
- 全員が「準備完了(Ready)」を宣言したら実行ステップへ進む

### 2-3. 実行(全員で順次処理) ✅

- 配置されたタスクを**1つずつ全員で**解決していく
- **処理順のルール:依存関係(親→子)を強制。依存上同列のタスクは PM が処理順を宣言する** ✅
- 各タスクの解決手順:
  1. 必要トークン数が積まれているか確認(不足ならこのフェーズは未解決のまま残る)
  2. 実行条件チェック(必要スキル・レベル、疲労制限)
  3. 秘匿要件ありの場合:要件カードを2枚引き、1枚を選んで適用 ✅
  4. 実行コスト(予算)を支払う
  5. 成果物トークン(Lv1/Lv2、0〜2個)を獲得
  6. 参加プレイヤーに実行時疲労を加算
  7. イベントマークがあればイベントカードを引いて解決
  8. 特殊効果(例:次タスクコスト減)を適用

### 2-4. フェーズ終了

1. **納期(D)判定**:未解決タスク数がプロジェクトシートの許容数を超えていれば CS 減少
2. **品質(Q)判定**:高品質成果物(Lv2)の数がフェーズの品質基準に達しているか判定。未達なら CS 減少 🔧
3. **疲労の自然回復**:全員 疲労 -1 ✅
4. 解決済みタスク上のトークンを回収。未解決タスクの扱いは ❓ TODO(暫定:**トークンは載せたまま次フェーズに持ち越し**)
5. フェーズ進行トラックを進める。第4フェーズ終了なら最終判定へ

## 3. QCD と CS の関係

### ✅ 確定している構造

- **C(コスト)= 予算トラック**:タスク実行コストで減少。枯渇すると追加請求が必要
- **D(納期)= フェーズ終了時の未解決タスク数判定**
- **Q(品質)= 高品質成果物数の基準判定**
- **CS(顧客満足度)= 唯一の勝敗トラック**。Q/D の未達や一部イベント、追加請求で増減する

### 🔧 暫定:クライアントカードの QCD 重み

クライアントカードは QCD の重みパラメータを持つ(例:C重視5/Q3/D2)。
暫定ルール:**各判定での CS 増減量に、クライアントの該当重みを乗算(または加算)する**。

> 例:納期超過のCS基本ペナルティが -1、クライアントのD重みが3なら CS -3。
> 「コスト重視クライアントに追加請求すると痛い」「品質重視クライアントは納期遅延に寛容」という非対称性を作るのが狙い。係数式は要プレイテスト。

### ✅ 追加請求

- 追加請求スペースにトークンを配置すると、**CS -1 と引き換えに予算を回復**
- 回復量(追加請求レート)はプロジェクトシートで定義 🔧(暫定:CS -1 あたり予算 +3)

## 4. 疲労システム 🔧

> 数値はすべて `GameConfig` の初期値。設定画面から変更可能にすること。

| 項目 | 暫定値 |
|---|---|
| 疲労レベル | Lv0〜Lv3 の4段階(初期 Lv0) |
| タスク実行時の疲労 | 通常タスク +1 / 重タスク +2(タスクタイルに定義) |
| Lv1 | ペナルティなし |
| Lv2 | 次フェーズの行動トークン補充 -1 |
| Lv3 到達時 | 即座に**限界イベントカード**を1枚引いて解決し、Lv2 に戻る |
| 休憩スペース | 行動トークン1個の配置で疲労 -2 |
| フェーズ終了時 | 全員 疲労 -1(自然回復) |

設計意図:2フェーズ連続で無理をすると Lv3 に届くテンポ。「休憩を挟む」か「限界イベントのギャンブル」かの判断が毎フェーズ生まれることを狙う。

### 限界イベントカード ✅

- 必ず発動するデメリット効果(予算減少、CS減少、Q低下 など)
- **「何も起きない」カードを1枚含む**(ギャンブル性の担保)

## 5. スキルと成長

### ✅ 確定している構造

- スキルは **3系統**、各 Lv1〜3(❓ 上限は要確認。暫定 Lv3)
- タスクの実行条件としてスキル系統×レベル(Lv1〜2)が要求される
- **学習タイル**にトークンを配置することでスキルアップ(またはスキルカード入手)できる
- 学習タイルは2〜3枚(❓ 毎フェーズ引くか、メインボード固定かは未決。暫定:**メインボード固定**)

## 6. カード・コンポーネント定義

### クライアントカード ✅(セットアップ時1枚)
- QCD 重みパラメータ(ランダムに異なる)
- 性格フレーバー:丸投げ系、細かい系、現場重視 など(❓ 性格がルール効果を持つかは TODO)

### プロジェクトカード ✅(セットアップ時1枚)
- プロジェクト種別:CMS、ブランドサイト、キャンペーン など
- 特殊要件:多言語、GA実装、レスポンシブ対応 など(❓ ルール効果の具体化は TODO)

### プロジェクトシート ✅
シナリオ定義。以下を含む:
初期QCD/CS/予算値、フェーズごとの品質基準・納期許容数・CS減少値、追加請求レート、特殊ルール、初期タスク構成、使用イベントデッキ(難易度調整)。

### タスクタイル ✅
| フィールド | 内容 |
|---|---|
| タスク名 | フレーバー(フェーズに対応した内容) |
| 必要行動トークン数 | 積み上げ式で充足 |
| 成果物獲得数 | 0〜2個(Lv1/Lv2) |
| スキル条件 | 系統×Lv1〜2 |
| 秘匿要件フラグ | あり→実行時に要件カード2枚から1枚選択 |
| 連鎖(依存)要件 | 親タスクの解決が前提 |
| 協業フラグ | ❓ 効果未定義(暫定:複数人のトークンが必須/ボーナス) |
| イベントマーク | あり→解決時にイベントカードを引く |
| 実行コスト | 予算消費値 |
| 実行時疲労値 | +1 または +2 |
| 特殊効果 | 例:次タスクコスト減 |

### イベントカード ✅
- 種類:要件変更、追加機能、納期変更、バグ報告、予算追加 など
- 効果:Q/C/D、疲労、スキル、予算への影響。発動条件(タイミング、対象プレイヤー、タスク種類)を持つ

### 要件カード ✅
- タスクの秘匿要件(本当の必要スキル など)
- 秘匿要件ありタスクの実行時に2枚引いて1枚選ぶ
- ❓ TODO:カードリストの具体内容

### 個人目標カード ✅
- 各自1枚、非公開
- 条件例:EP◯点、特定成果物◯個、QCDのうち1つ特化 など

## 7. セットアップ ✅

1. プロジェクトカード・クライアントカードを各1枚公開
2. プロジェクトシートに従い QCD・CS・予算トラックを初期化
3. イベントデッキを構成しシャッフル
4. 第1フェーズのタスクタイルを公開し、PM 主導で依存関係を考慮して配置
5. 各プレイヤーにロール・個人ボードを配布。個人目標カードは `personalGoalChoices`(🔧 暫定2)枚ずつ配り、各自1枚を秘密裏に選ぶ(v2.1 §11。1枚配布設定なら従来どおり自動割当)。マイルストーンカードを `milestoneCount`(🔧 暫定3)枚公開する
6. ロールに応じた初期スキル値を設定
7. 行動トークンを配布(🔧 初期値:各5個)

## 10. 炎上システム 🔧(v2.1。借用元:パンデミック)

> 数値はすべて `GameConfig` の初期値。`fireEnabled: false` で v1 ルールに戻せる。

- **炎上デッキ**:各タスクタイルを指す対象カード + 大炎上カード `epidemicCount` 枚で構成。
  フェーズ開始時にそのフェーズのタスクカードを山に混ぜてシャッフルする
- フェーズ開始時(タスク公開後・イベント前)に炎上カードを `firePerPhase` 枚引き、
  対象タスクに🔥トークンを1個置く。対象が盤面にない・解決済みなら何も起きない
- **🔥1個につき、そのタスクの必要行動トークン +1**(スコープクリープ)
- **延焼**:同一タスクに `fireOutbreakThreshold` 個目の🔥は置けず、代わりに
  盤面上の未解決の子タスク全員に🔥1個ずつ + **CS -`fireOutbreakCsPenalty`**。
  子がいなければ同フェーズの他の未解決タスクへ。連鎖延焼あり(同一連鎖内で同じタスクは1回まで)
- **大炎上**:炎上捨て札を山に戻してシャッフルし、PM が選んだ未解決タスクに🔥2個。
  大炎上カードはゲームから除外する
- **消火**:プランニング中、行動トークン `extinguishCost` 個で任意のタスクの🔥を1個除去する
- 未解決タスクの🔥は次フェーズに持ち越す(燃え続ける)

### ❓ 未決(プレイテストで決める)
- 解決時に🔥が残っていた場合の追加ペナルティ(暫定:なし。必要トークン増のみ)
- 消火を実行ステップ中にも許可するか(暫定:プランニング中のみ)

## 11. EP とマイルストーン 🔧(v2.1。借用元:白ブラス + テラフォーミングマーズ)

### EP(Engagement Point)

> **EP = 自分の仕事が他人に使われた回数。** 公開トラック。

EP が +1 される契機(`epEnabled` 時):

1. 自分が参加した**親タスク**の子タスクを、自分が参加していない形で他プレイヤーが解決した
2. 自分が**消火**したタスクが、そのフェーズ中に自分の参加なしで解決された

(§3 資源・§4 産出の導入後に契機を追加予定 → rules-v2-proposal.md 参照)

### マイルストーン(公開・早取り)

- セットアップ時に `milestoneCount` 枚公開。**最初に条件を達成した1人だけ**が獲得する
- 同時達成はアクション解決順で先の者
- 例:火消し番長(累計消火5回)/ 品質職人(Lv2成果物2個に関与)/ 学習の鬼(スキルアップ2回)/
  縁の下(EP4)/ 鬼の工数(1フェーズにトークン5個以上をタスクへ配置)

### 個人目標の選択制

- セットアップ時に各自 `personalGoalChoices` 枚から1枚を秘密裏に選ぶ(選ばなかった札は公開しない)

### 個人勝利条件(§1 の更新)

> 個人勝利 = チーム勝利 かつ(**秘匿の個人目標達成** または **マイルストーン獲得**)

チーム敗北時に個人勝利がないのは従来どおり。
