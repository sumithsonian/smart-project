# ルール v3 提案 — 「当てはめ」から「配属」へ

> **このファイルは提案(discussion draft)であり、ルールの正ではない。**
> v2.2(配属トリアージ)のプレイテストで「行動トークンをただタスクに当てはめるだけで、
> ジレンマがなく終わる」という致命的フィードバックが出た。本提案はその根治を狙う全面改訂案。
>
> 章ごとに採否・数値を議論し、確定した章から [RULES.md](RULES.md) に反映 → GameConfig フラグ付きで実装する。

## 0. 設計の前提(ヒアリング確定済み)

| # | 前提 | 備考 |
|---|---|---|
| 1 | **プロジェクトの追体験が最優先**(教育ボドゲ) | すべての採否判断はここに従う。メカニクスの面白さより体験の忠実さ |
| 2 | 配置の動詞は「人を割り当てる」(A案) | 「当てはめ作業」批判の根因が均質トークンの無限分割にあるため |
| 3 | 重さはミドル級(60〜90分、パンデミック級) | ヘビー級の資源変換系は入れない |
| 4 | **物理版で成立するルールのみ採用(必須)** | デジタル版はあくまでアナログ検証用。自動計算・個別秘匿依存の仕掛けは不可 |
| 5 | 面白さの核は「修羅場のトリアージ」+「チーム編成と育成」 | 供給<需要、スキルマッチング、学習は痛い投資 |
| 6 | **だまし合い(裏切り・欺瞞)の演出は不要** | ただし「最終決定は本人」の同時コミットは自己決定の表現として**採用**(確認済み)。対立煽り系の個人目標は不採用 |

## 1. 診断サマリ(定量監査 + 構造批評)

### 定量(コンテンツ実数の集計)

| 項目 | 実数 | 判定 |
|---|---|---|
| トークン需給 | 供給80 vs 実需59(🔥込み)= **余剰+21** | 🚨 全部やってもまだ余る。ジレンマの物理的不在 |
| 予算収支 | 初期18 vs コスト計25 = **赤字-7**(イベント期待値±0) | ✅ 良い圧力。追加請求が調整弁として機能 |
| 品質 | Lv2入手可能10個 vs 基準1×4フェーズ | ⚠️ 緩い。品質は実質フリー達成 |
| スキルゲート | Lv2ゲート5つ中3つを**エンジニアが独占** | ⚠️ 非対称は良いが1人に集中しすぎ |
| 疲労 | P3で限界イベントほぼ確定 | ✅ 曲線は機能。ただし他システムから孤立 |

### 構造(サブシステム監査の結論)

- **死に荷重**:協業フラグ(効果未定義)/ 秘匿要件(配置判断の後にしか効かない)/ EP(受動指標で意思決定に介入しない)/ 学習v1(配置と資源競合しない)
- **ジレンマ欠落の構造要因**(重要度順):
  1. 供給過多 × トークン均質性(誰が置いても同じ・無限分割可能)
  2. 手番なし同時配置による奉行問題(1人が全体最適を計算して指示できる)
  3. フェーズ内完結・非永続性(長期計画が不要、毎フェーズが独立した線形計画)
  4. 痛みの可視化遅延(諦めの代償がフェーズ終了の一括判定でしか見えない)
  5. 配置段階の情報が完全公開で、駆け引き・腹の探り合いがゼロ
- **残す骨格**:タスク依存ツリー+PM裁定 / QCD→CS単一勝敗軸+クライアント重み / 秘匿個人目標×チーム勝利の準協力構造

## 2. 設計原則(v3 の憲法)

1. **配置は「人を仕事に就ける」**。不可分(薄く配れない)・排他(ここに立てばあそこに立てない)・固有(誰が立つかで結果が変わる)
2. **供給 < 需要**。全部はできない。毎フェーズ何かを諦める
3. **痛みは配置の瞬間に見える**。遅延した一括清算ではなく、盤上で刻々と膨らむ
4. **脅威はメモリ付き・可視・対処可能**(パンデミック原則。理不尽な被弾にしない)
5. **足す前に削る**。死に荷重4つを削って、複雑性予算を新しい核に回す

## §1 週次ワーカーコミット(コア。ユーザー発案の週次ラウンド制 + ワーカープレイスメント)

**狙い**:構造要因1(均質性)・2(奉行)・3(非永続性)を一つの仕組みで根治する。
行動トークンを廃止して配置の単位を「プレイヤー本人」にし、さらに**フェーズを週(ラウンド)に分割**して、
週と週の間に盤面が変わるようにする。

> 元になった着想(プレイテストからのフィードバック):
> 「フェーズ中に盤面が変わらないのが原因では。一列(一週)を1ラウンドにして、
> 列ごとに一斉配置 → 次の列に行く前にイベントで盤面を変えれば、計画が崩れていい」

### 構造:フェーズ = 月、ラウンド = 週

1フェーズ = `roundsPerPhase`(暫定3)週。各週は以下の3ステップ:

1. **週初のトラブル**:炎上カードを `firePerRound`(暫定1)枚引く + 週次イベント(要件フリップ等)を解決する。
   **盤面がここで変わる**(🔥で必要席が増える、専門席の系統が変わる、誰かが離脱する…)
2. **朝会(同時配置)**:全員で相談しながら、各自「今週の自分」を同時に決める:
   - **タスクの席**に立つ(自分のスキルを持ち込む)
   - **学習**(今週は現場に立たない。フェーズ終了時に選んだ系統 +1Lv)
   - **休憩**(今週は現場に立たない。疲労 -`restRecovery`)
   - **消火**(任意のタスクの🔥1個を除去)
   - さらに任意で**残業**(疲労 +`overtimeFatigue` を即時に受けて、上記をもう1枠)。
     疲労Lv2 以上は残業禁止
3. **週末の解決**:全席が埋まっているタスクを解決する(親が未解決のタスクは解決できない)。
   複数あるときの順序は PM が決める(全フェーズ分の処理順宣言は廃止。今週ぶんだけの軽い裁定になる)

フェーズ終了(= `roundsPerPhase` 週の終わり)に品質判定・締切利息(§3)・疲労自然回復・タスク持ち越し。

### なぜジレンマが生まれるか

- **供給**:4人 × 3週 = 12人週/フェーズ(+残業。ただし健康を支払う)
- **需要**:タスク5〜6枚 × 席1〜2 + 週次の🔥 + 学習の誘惑 + 休憩の必要 = **常時12人週超**
- **週間の断絶が計画を分割する**:週初のトラブルで盤面が変わるため、フェーズ全体の最適解は
  そもそも計算できない。「先週立てた配属が今週崩れる」= 奉行は成立せず、だまし合いも不要
- **依存レーンと時間が一致する**:起点タスクを今週解決しないと、依存タスクは来週まで着手できない。
  「今週どれを進めるか」が納期(締切利息)への布石になる

### 既存ルールへの影響

- `PLACE_TOKEN` / `RETRIEVE_TOKEN` / `tokensPerPhase` を廃止(フラグオフで従来動作)
- `DECLARE_TASK_ORDER`(フェーズ全体の処理順宣言)を廃止。週末に複数タスクが解決可能な場合のみ PM が順序を裁定
- 学習は「トークンN個」から「週の拘束」へ(1週=+1Lv だと軽すぎる場合、Lv2 昇格は2週に。要テスト)
- 休憩・消火は週の選択肢に統合(独立スペース廃止)。追加請求は PM 裁量のフリーアクション(フェーズ `extraBillingPerPhase` 回まで)
- 炎上ドローは「フェーズ開始に2枚」から「毎週1枚」へ(総量は微増。脅威が分散して蓄積が見える)

## §2 席(シート)とタスク定義の改訂

**狙い**:構造要因1の後半(専門家の無限分割)の根治 + 死に荷重2つの吸収。

### ルール案

1. タスクの `requiredTokens` を廃止し、**席(1〜2席)** に置き換える:
   - **専門席**:系統×Lv 指定。満たすプレイヤーが立てば充足。**やっつけ**(v2.2 実装済み)で無資格でも立てるが、成果物ダウン+疲労+CS債務
   - **人手席**:誰でも立てる
   - 全席が埋まった状態で週末(§1 ステップ3)を迎えたら解決
2. **協業フラグは削除**し、「2席タスク」として表現(協業=席が2つあるタスク。定義済みの効果を持つ)
3. **秘匿要件(実行時の受動ドロー)は削除**し、**要件フリップ**に統合:
   - 要件フリップ = フェーズ開始イベントの一種。「対象タスクの専門席の系統が変わる」
   - 配置**前**に判明する(スコープ変更として計画を壊す)か、配置**後**の実行前に判明する(現場の悲鳴)かは `flipTiming` で切り替えてテスト
4. Lv2ゲートの配分を再設計:各ロールが最低2つの専門席で「唯一の適任」になるようコンテンツを調整(現状のエンジニア3独占を分散)

### 過剰スペック・外注(v2.2 実装済み)はそのまま接続

- Lv2 の人が Lv1 席に立てば実行コスト割引
- 外注は「席を金で埋める」(予算+CS)。回数制限あり

## §3 締切利息(痛みの前倒し。借用元:アグリコラの物乞い + パンデミックの都市キューブ)

**狙い**:構造要因3(非永続性)と4(痛みの遅延)の同時解決。

### ルール案

1. 各タスクタイルに**締切フェーズ**を刷る(基本 = 登場フェーズ。余裕のあるタスクは +1)
2. **締切を過ぎた未解決タスクは、毎フェーズ終了時に CS -`deadlineInterestCs`(暫定1)を刻む**。
   物理版はタスクに遅延マーカーを置くだけ(可視の債務)
3. 現行の「フェーズ終了時の未解決数 vs 許容数」の一括納期判定は**廃止**(利息方式に一本化。システム数は増えない)
4. 🔥は従来どおり持ち越し・延焼(遅延タスクは燃えやすい盤面になり、脅威が自然に蓄積する)

### 効果

- 「このタスクを見送る」と決めた瞬間、**いつまでにいくら払うことになるか**が盤上に見える
- 「今フェーズ捨てて来フェーズ拾う」という**フェーズをまたぐ計画**が初めて意味を持つ
- クライアントD重みが利息に乗る → 納期重視クライアントでは放置が本当に痛い

## §4 朝会プロトコルと週次トラブル

### 4-1. 朝会の同時コミット + PM 調整(奉行対策の仕上げ)

**狙い**:配属の最終決定権を本人に置く(だまし合いではなく**自己決定**の表現)。
週次ラウンド制(§1)で「全体最適の計算」を壊し、本プロトコルで「指示の実行」を壊す。二段構え。

1. 朝会の相談は自由(むしろ推奨。ここがこのゲームの一番楽しい時間のはず)
2. 相談が済んだら、**各自「今週の自分」を伏せて選び、一斉公開**する
   (物理版:個人ボードに担当カードを裏向きに置き、せーので表にする)
3. 公開後、**PM は1枠だけ担当を変更できる**(`pmAdjustmentsPerWeek`、暫定1)。
   マネジメントの見せ場であり、事故(2人が同じ席に被る等)の救済弁でもある
4. 残業の宣言は公開後に順番に行う(残業は「穴が見えてから埋める」動きなので公開でよい)

- 効果:奉行は「提案」はできても「決定」はできない。個人目標(§5)の働き方の個性が、
  伏せた1手に自然に滲む(裏切りではなく「私はこうしたい」の表明)
- ❓ PM 調整は強権(本人の同意不要)か協議(本人が拒否可)か → 暫定:**強権**。
  「PMに現場を動かされる」もまたプロジェクトの追体験のうち。荒れるようなら協議に緩める

### 4-2. 週次トラブルのデザイン

**狙い**:週初に引く「トラブル」の中身を、**配属計画を壊す方向**に設計する。
原則は「メモリ付き・可視・対処可能」(パンデミック原則)。理不尽な即死ではなく、
「先週の計画がこう崩れた。さあどう組み替える?」を作る。

### トラブルの型(週次イベントの設計指針)

| 型 | 例 | 壊れるもの |
|---|---|---|
| スコープ増 | 🔥(必要席にカウントされる工数+1) | 「1人で足りる」の見込み |
| 要件フリップ | 専門席の系統が変わる(デザイン席→エンジ席) | 「適任を当てた」の前提 |
| 稼働離脱 | 指名プレイヤーが来週使えない(私用・体調) | 「来週あの人に頼む」の予定 |
| 締切前倒し | 対象タスクの締切フェーズ -1 | 「後回しで良い」の判断 |
| 追い風 | 予算追加・タスク1つのコスト減(良いことも起きる) | (救済。全部が悪意だと学習教材として歪む) |

- 週次イベントデッキは炎上デッキと同様、**捨て札リシャッフルでメモリを持たせる**(一度出たトラブルはまた来やすい)
- 教育観点の意図:トラブルは「罰」ではなく**「再計画の練習問題」**。だから必ず対処手段(消火・外注・やっつけ・組み替え)が存在する型だけを入れる

## §5 個人目標 = 「働き方の個性」(だまし合いではなく)

**狙い**:死に荷重化している個人目標を復活させる。ただし前提6により、**チームを裏切る誘惑ではなく
「自分のキャリア・働き方のこだわり」**として設計する。教育的には「メンバーそれぞれに事情がある中で
チームを回す」の追体験になる。

### 条件の設計方針(例)

| タイプ | 例 | 生まれる個性(裏切りではない) |
|---|---|---|
| 専門性 | 自分の系統の専門席に3回立つ | 「本職の仕事がしたい」志向 |
| 成長 | 学習を2回行う | 「この案件で腕を上げたい」若手 |
| 健康 | 疲労Lv2 以上に一度もならない | ワークライフバランス派 |
| 品質 | 自分が席に立ったタスクから Lv2 を2個 | 職人気質 |
| 縁の下 | 自分が解決に関わった親タスクの子を、他人が解決 3回 | 下準備・仕組み化の人 |

- 目標は**公開してもゲームが壊れない**強度にする(秘匿は「ちょっとした自分らしさ」の演出に留める。
  物理版で公開プレイしても成立すること)
- 効果:全員の最適解が微妙にズレるので、朝会で「私はこっちに行きたい」という**本人の意思**が自然に出る。
  奉行対策の補助線にもなる(命令ではなく交渉になる)
- **EP トラックは廃止**。「縁の下」系は上記のような直接カウントに書き換え(集計は同じ、公開トラックの管理コストだけ消える)
- マイルストーン(公開早取り)は残す。条件を上記と同系統の行動ベースに更新

## §6 削除・統合リスト(複雑性収支)

| 削るもの | 行き先 |
|---|---|
| 行動トークン(均質資源) | §1 週次ワーカーコミット |
| 協業フラグ(未定義) | §2 の2席タスク |
| 秘匿要件の実行時ドロー | §2/§4 の要件フリップ・トラブル |
| EP トラック | §5 の行動ベース個人目標 |
| 学習タイルのトークン積み | §1 の「学習に週を充てる」 |
| フェーズ終了の一括納期判定 | §3 の締切利息 |
| フェーズ全体の処理順宣言(DECLARE_TASK_ORDER) | §1 の週末解決(PM は今週ぶんだけ裁定) |

**収支:7システム削除・置換に対し、新規概念は「席」「週(ラウンド)」「残業」「締切利息」「朝会コミット」の5つ。**
ルール総量はほぼ横ばいで、全システムがコアループ(今週、誰がどこに立つか)に接続される。

## §7 GameConfig 追加・変更案(暫定値)

```
// §1 週次ワーカーコミット
workerCommitEnabled: true
roundsPerPhase: 3           // 1フェーズの週数
firePerRound: 1             // 週初に引く炎上カード枚数(firePerPhase を置換)
overtimeFatigue: 1          // 残業の即時疲労
overtimeMaxPerWeek: 1       // 1人・1週あたりの残業枠
noOvertimeAtFatigueLv: 2    // このLv以上は残業禁止
extraBillingPerPhase: 1     // 追加請求のフェーズ回数上限

// §2 席
seatsEnabled: true          // requiredTokens → 席 モデル
learnWeeksPerLevel: 1       // +1Lv に必要な学習週数(Lv2昇格は2週の案あり)

// §3 締切利息
deadlineInterestEnabled: true
deadlineInterestCs: 1       // 超過タスク1枚・1フェーズあたりのCS(D重み適用)
defaultDeadlineOffset: 0    // 締切 = 登場フェーズ + この値

// §4 朝会コミット + 週次トラブル
standupCommitEnabled: true  // 伏せて選ぶ→一斉公開(オフなら公開相談のみ)
pmAdjustmentsPerWeek: 1     // 公開後に PM が動かせる担当数
flipTiming: "week_start"    // 要件フリップの発火(week_start | pre_execution)

// §5(EP廃止)
epEnabled: false            // 廃止(後方互換のためフラグは残す)
```

## §8 導入ロードマップ

| 段階 | 導入 | 検証したいこと |
|---|---|---|
| v3.0 | §1 週次ワーカーコミット + §2 席(+v2.2接続) | 週ごとの再計画が楽しいか。「今週、誰をどこに」が苦渋になるか。残業と疲労のループ |
| v3.1 | §3 締切利息 + §4-2 トラブル型の拡充 + §6 削除断行 | 諦めの痛みが配置時に見えるか。週次トラブルで計画が「良い感じに」崩れるか |

各段階はフラグでオン/オフでき、オフなら前段階のルールで動く(リグレッションテスト維持)。

## §9 未決事項

- ❓ 週数(`roundsPerPhase`)は3か2か(3週×4フェーズ=12朝会。プレイ時間60〜90分に収まるかで判断)
- ❓ 席が埋まったタスクの解決タイミング:週末一括か、埋まった瞬間か(暫定:週末一括。物理版の処理が楽)
- ❓ 残業枠は週1で足りるか(2つ許すと供給が緩む。疲労コストを2倍にして許可する案も)
- ❓ 席の総数の適正値(`demandSeatRatio` 1.3〜1.5 をボットシミュレーションで検証)
- ❓ PM 調整は強権か協議か(§4-1。暫定:強権)
