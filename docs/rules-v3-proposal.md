# ルール v3 提案 — 「当てはめ」から「配属」へ

> **このファイルは提案(discussion draft)であり、ルールの正ではない。**
> v2.2(配属トリアージ)のプレイテストで「行動トークンをただタスクに当てはめるだけで、
> ジレンマがなく終わる」という致命的フィードバックが出た。本提案はその根治を狙う全面改訂案。
>
> 章ごとに採否・数値を議論し、確定した章から [RULES.md](RULES.md) に反映 → GameConfig フラグ付きで実装する。

## 0. ヒアリング仮定(要確認)

設計の分岐点になる好みを以下のとおり仮定した。**違っていたら言ってほしい。章の組み替えで対応する。**

| # | 仮定 | 根拠 |
|---|---|---|
| 1 | 配置の動詞は「人を割り当てる」(A案) | 「当てはめ作業」批判の根因が均質トークンの無限分割にあるため |
| 2 | 重さはミドル級(60〜90分、パンデミック級) | 現行設計の自然な着地点。ヘビー級の資源変換系は入れない |
| 3 | 物理版で成立するルールのみ採用 | 当初目的が「物理版に向けたルール検証」のため。自動計算・個別秘匿依存の仕掛けは避ける |
| 4 | 面白さの核は「修羅場のトリアージ」+「チーム編成と育成」 | これまでの議論(供給<需要、スキルマッチング、学習は痛い投資)の延長 |

## 1. 診断サマリ(定量監査 + 構造批評)

### 定量(コンテンツ実数の集計)

| 項目 | 実数 | 判定 |
|---|---|---|
| トークン需給 | 供給80 vs 実需59(🔥込み)= **余剰+21** | 🚨 全部やってもまだ余る。ジレンマの物理的不在 |
| 予算収支 | 初期18 vs コスト計25 = **赤字-7**(イベント期待値±0) | ✅ 良い圧力。追加請求が調整弁として機能 |
| 品質 | Lv2入手可能10個 vs 基準1×4フェーズ | ⚠️ 緩い。品質は実質フリー達成 |
| スキルゲート | Lv2ゲート5つ中3つを**エンジニアが独占** | ⚠️ 非対称は良いが1人に集中しすぎ |
| 疲労 | P3で限界イベントほぼ確定 | ✅ 曲線は機能。ただし他システムから孤立 |

### 構造(サブシステム監査の結論)

- **死に荷重**:協業フラグ(効果未定義)/ 秘匿要件(配置判断の後にしか効かない)/ EP(受動指標で意思決定に介入しない)/ 学習v1(配置と資源競合しない)
- **ジレンマ欠落の構造要因**(重要度順):
  1. 供給過多 × トークン均質性(誰が置いても同じ・無限分割可能)
  2. 手番なし同時配置による奉行問題(1人が全体最適を計算して指示できる)
  3. フェーズ内完結・非永続性(長期計画が不要、毎フェーズが独立した線形計画)
  4. 痛みの可視化遅延(諦めの代償がフェーズ終了の一括判定でしか見えない)
  5. 配置段階の情報が完全公開で、駆け引き・腹の探り合いがゼロ
- **残す骨格**:タスク依存ツリー+PM裁定 / QCD→CS単一勝敗軸+クライアント重み / 秘匿個人目標×チーム勝利の準協力構造

## 2. 設計原則(v3 の憲法)

1. **配置は「人を仕事に就ける」**。不可分(薄く配れない)・排他(ここに立てばあそこに立てない)・固有(誰が立つかで結果が変わる)
2. **供給 < 需要**。全部はできない。毎フェーズ何かを諦める
3. **痛みは配置の瞬間に見える**。遅延した一括清算ではなく、盤上で刻々と膨らむ
4. **脅威はメモリ付き・可視・対処可能**(パンデミック原則。理不尽な被弾にしない)
5. **足す前に削る**。死に荷重4つを削って、複雑性予算を新しい核に回す

---

## §1 ワーカーコミット(コア。借用元:ワーカープレイスメント全般 + アグリコラの「家族」)

**狙い**:構造要因1(均質性)の根治。行動トークンを廃止し、配置の単位を「プレイヤー本人」にする。

### ルール案

毎フェーズ、各プレイヤーは以下を選ぶ:

1. **本業(主担当)を1つ**:
   - **タスクの席**に立つ(自分のスキルを持ち込む)
   - **学習**(このフェーズは現場に立たない。次フェーズ開始時に選んだ系統 +1Lv)
   - **休憩**(現場に立たない。疲労 -`restRecovery`)
2. **任意で残業(疲労 +`overtimeFatigue` を即時に受ける)**:
   - **別タスクの席**に立つ(2箇所目)
   - **消火**(任意のタスクの🔥1個を除去)
   - **追加請求の営業対応**(チームでフェーズ1回まで。CS-1 で予算 +`extraBillingBudget`)

- **疲労Lv2 のプレイヤーは残業禁止**(トークン-1ペナルティの置き換え。疲労がコアループに直結する)
- 疲労Lv3 到達で限界イベント(従来どおり)

### 需給の設計(なぜジレンマが生まれるか)

- 供給:主担当4枠 + 残業0〜4枠(ただし残業は健康を支払う)= **実質4〜8枠**
- 需要:タスク5〜6枚 × 席1〜2 + 消火 + 学習の誘惑 + 休憩の必要 = **常時8〜12枠ぶん**
- **「全員残業すれば回る。でも全員残業したら P3 で限界イベントの連鎖が来る」**——働き方そのものが毎フェーズの賭けになる。疲労が「独立メーター」から「中央銀行」に変わる

### 既存ルールへの影響

- `PLACE_TOKEN` / `RETRIEVE_TOKEN` / `tokensPerPhase` を廃止(フラグオフで従来動作)
- 学習は「トークンN個」から「1フェーズ拘束」へ(v2提案 §2 の trainingTurns 方式に一本化)
- 休憩・追加請求・消火はすべて上記メニューに統合(独立スペース廃止)

## §2 席(シート)とタスク定義の改訂

**狙い**:構造要因1の後半(専門家の無限分割)の根治 + 死に荷重2つの吸収。

### ルール案

1. タスクの `requiredTokens` を廃止し、**席(1〜2席)** に置き換える:
   - **専門席**:系統×Lv 指定。満たすプレイヤーが立てば充足。**やっつけ**(v2.2 実装済み)で無資格でも立てるが、成果物ダウン+疲労+CS債務
   - **人手席**:誰でも立てる
   - 全席が埋まった状態で実行ステップを迎えたら解決
2. **協業フラグは削除**し、「2席タスク」として表現(協業=席が2つあるタスク。定義済みの効果を持つ)
3. **秘匿要件(実行時の受動ドロー)は削除**し、**要件フリップ**に統合:
   - 要件フリップ = フェーズ開始イベントの一種。「対象タスクの専門席の系統が変わる」
   - 配置**前**に判明する(スコープ変更として計画を壊す)か、配置**後**の実行前に判明する(現場の悲鳴)かは `flipTiming` で切り替えてテスト
4. Lv2ゲートの配分を再設計:各ロールが最低2つの専門席で「唯一の適任」になるようコンテンツを調整(現状のエンジニア3独占を分散)

### 過剰スペック・外注(v2.2 実装済み)はそのまま接続

- Lv2 の人が Lv1 席に立てば実行コスト割引
- 外注は「席を金で埋める」(予算+CS)。回数制限あり

## §3 締切利息(痛みの前倒し。借用元:アグリコラの物乞い + パンデミックの都市キューブ)

**狙い**:構造要因3(非永続性)と4(痛みの遅延)の同時解決。

### ルール案

1. 各タスクタイルに**締切フェーズ**を刷る(基本 = 登場フェーズ。余裕のあるタスクは +1)
2. **締切を過ぎた未解決タスクは、毎フェーズ終了時に CS -`deadlineInterestCs`(暫定1)を刻む**。
   物理版はタスクに遅延マーカーを置くだけ(可視の債務)
3. 現行の「フェーズ終了時の未解決数 vs 許容数」の一括納期判定は**廃止**(利息方式に一本化。システム数は増えない)
4. 🔥は従来どおり持ち越し・延焼(遅延タスクは燃えやすい盤面になり、脅威が自然に蓄積する)

### 効果

- 「このタスクを見送る」と決めた瞬間、**いつまでにいくら払うことになるか**が盤上に見える
- 「今フェーズ捨てて来フェーズ拾う」という**フェーズをまたぐ計画**が初めて意味を持つ
- クライアントD重みが利息に乗る → 納期重視クライアントでは放置が本当に痛い

## §4 同時秘密コミット(奉行対策。借用元:王と枢機卿の同時プロット + 提案v2 §4 の「最終決定は持ち主」)

**狙い**:構造要因2(奉行)と5(駆け引き不在)。段階導入(v3.1 以降)。

### ルール案

1. プランニングの相談は自由(むしろ推奨)。ただし**主担当の最終決定は各自が秘密に選び、一斉公開**する
   (物理版:役割ボードに担当カードを伏せて置き、全員で表にする)
2. 公開後、**調整ラウンドを1回だけ**許す:全員同意なら1人だけ担当を変更できる(現場の融通)
3. 残業の宣言は公開後に順番に(残業は「穴が見えてから埋める」動きなので公開でよい)

### 効果

- 奉行が全員分を指示できなくなる(最後の1手は本人のもの)
- 個人目標(§5)との組み合わせで「あの人、なぜ品質タスクに行かなかった…?」という**腹の探り合い**が発生する
- 純協力の相談の楽しさは調整ラウンドで担保

## §5 個人目標の対立設計(準協力の再生)

**狙い**:死に荷重化している個人目標を、チーム行動と**軽く衝突**させて復活させる。EP は廃止し、行動ベースの条件に置き換える。

### 条件の設計方針(例)

| タイプ | 例 | チームとの衝突 |
|---|---|---|
| 独占欲 | 自分が主担当したタスクから Lv2 成果物3個 | おいしいタスクを譲らない誘惑 |
| 働き方 | 残業を一度もせずチーム勝利 | 修羅場で手を貸さない誘惑 |
| 政治 | 追加請求を2回、自分が営業対応する | CSを削る行動を自分から提案する誘惑 |
| 育成 | 学習を2回行う(=現場を2フェーズ離脱) | 猫の手も借りたい時に学校に行く誘惑 |
| 火消し | 消火を4回(=残業4回=疲労を抱える) | 健康と引き換えの英雄願望 |

- **EP トラックは廃止**。「縁の下」系の目標は「自分が立った親タスクの子を他人が解決 N回」のような直接カウントに書き換え(集計は同じ、公開トラックの管理コストだけ消える)
- マイルストーン(公開早取り)は残す。条件を上記と同系統の行動ベースに更新

## §6 削除・統合リスト(複雑性収支)

| 削るもの | 行き先 |
|---|---|
| 行動トークン(均質資源) | §1 ワーカーコミット |
| 協業フラグ(未定義) | §2 の2席タスク |
| 秘匿要件の実行時ドロー | §2 の要件フリップ・イベント |
| EP トラック | §5 の行動ベース個人目標 |
| 学習タイルのトークン積み | §1 の「学習に1フェーズ充てる」 |
| フェーズ終了の一括納期判定 | §3 の締切利息 |

**収支:6システム削除・置換に対し、新規概念は「席」「残業」「締切利息」「秘密コミット」の4つ。**
ルール総量はほぼ横ばいで、全システムがコアループ(誰がどこに立つか)に接続される。

## §7 GameConfig 追加・変更案(暫定値)

```
// §1 ワーカーコミット
workerCommitEnabled: true
overtimeFatigue: 1          // 残業の即時疲労
overtimeMaxPerPhase: 1      // 1人あたり残業枠
noOvertimeAtFatigueLv: 2    // このLv以上は残業禁止
extraBillingPerPhase: 1     // 追加請求のフェーズ回数上限

// §2 席
seatsEnabled: true          // requiredTokens → 席 モデル
flipTiming: "phase_start"   // 要件フリップの発火(phase_start | pre_execution)

// §3 締切利息
deadlineInterestEnabled: true
deadlineInterestCs: 1       // 超過タスク1枚・1フェーズあたりのCS(D重み適用)
defaultDeadlineOffset: 0    // 締切 = 登場フェーズ + この値

// §4 同時秘密コミット(v3.1)
secretCommitEnabled: false
adjustmentRounds: 1         // 公開後の調整回数

// §5(EP廃止)
epEnabled: false            // 廃止(後方互換のためフラグは残す)
```
