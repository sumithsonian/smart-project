# ルール v4 コア — 工数モデルによる統合(討議ドラフト)

> **位置づけ**:v3 までの「付け足し」を清算し、少ないルールで首尾一貫させる再設計。
> 本ドラフトが承認されたら、RULES.md を全面改訂し、エンジンは v4 として組み直す
> (v3 系フラグの分岐を温存せず、きれいに)。
>
> 設計のゴール(確認済みの前提):
> - 主眼は**教育**。「WBSを立て→役割を判断し→イベントを捌き→タスクをこなし→責任を果たし→お客様の成果に繋げる」の追体験
> - ロック型スキル(できる/できない)を**勾配型(誰でもできる、上手い人は速い)**に変え、配置の決定論を壊す
> - だまし合いなし。物理版で成立すること

## 0. たった1つの通貨:工数キューブ

このゲームのほぼすべては**工数キューブ**で説明される。

- タスクには **必要工数(系統つき)** が刷ってある(例:「トップページ実装 ⚙4」)
- プレイヤーがタスクに1週座ると、**その系統の自分のスキル値ぶん**のキューブを積む
  (⚙2 のエンジニアなら2個/週、⚙1 のディレクターなら1個/週、⚙0 の人は座れない)
- キューブは**タスクに残る**。人は毎週手元に戻り、続けるか離れるかを選び直す(拘束なし)
- 複数人で座れば足し算(2+1=3個/週)
- **必要数に達したら納品可能**。納品時にそのまま止めれば **Lv1**、
  さらに **+`qualityOvershoot`(暫定2)個**積み増してから納品すれば **Lv2**(品質は積み増しの意思)
- 🔥(炎上)1個 = 必要工数 +1(スコープクリープ。同じ通貨)
- 学習 = 自分のスキル+1(= 恒久的に速くなる。価値が数字で見える)

> これにより v3 の「専門席/人手席」「やっつけ+CS債務」「過剰スペック割引」「収穫の選択」
> 「品質閾値の別判定」の**5システムを削除**する。

## 1. ゲームの流れ

**1ゲーム = 4フェーズ(企画/設計/開発/テスト)、1フェーズ = 3週。**

### 1-1. スコープ会議(フェーズ開始。= 検収条件に約束し、WBS を立てる)

> **原則:タスクの完了は目的ではない。成果が目的。**
> お客様が求めるのは検収条件(プロダクトボードの状態)であり、タスクはそれを満たす**手段の選択肢**にすぎない。

1. **検収条件カード**を公開する(クライアント由来。例:「トップページが Lv2 で存在する」
   「CMS で更新できる」「全ページレスポンシブ」)。各条件はプロダクトボードのスロットと Lv を指す
2. チームで相談し、**今フェーズ満たす検収条件を選んで約束(コミット)**する。最終決定は PM 帽子(§3)
3. 約束した条件を満たすための**タスクを候補プールから選び、依存レーンに配置**する(= WBS)
   - レーン文法は固定:「起点 → 中盤 → 仕上げ」。カードに刷られた列にしか置けず、
     中盤・仕上げは**前列に1枚以上あること**(カード個別の依存チェーンは廃止)
   - **同じスロットを埋める手段は複数ある**(既製テーマで安く早く Lv1 上限/フルスクラッチで重いが Lv2 可、など)。
     **やらなくていいタスクが常に存在する**
4. **見積もりの責任**:約束した検収条件が未達のままフェーズ末を迎えるたび、**CS -`commitPenaltyCs`**
   (ビッドと締切利息はこの1ルールに統合。約束していない条件に罰はない——ただし最終検収(§2)には響く)

> チームの週間ベロシティに対して、検収条件の総需要は常に過剰にする。**全部は約束できない。**
> どの成果を約束し、どの手段で行き、何を見切るかがこの会議の本体。

### 1-2. 週(×3)

1. **週初のトラブル**:炎上カード1枚(🔥+1 or 大炎上)+ イベントカード1枚。イベントには
   **差し込みタスク**(計画外の新規需要)を含む——これが需要の動性を作る(パンデミックの「湧き」):
   - **手戻り**:納品済みスロットに手戻りキューブ(再度積まないと Lv ダウン/検収不成立)
   - **バグ報告**:対応タスクが場に出る。放置すると毎フェーズ CS-1、対応すれば止まる
   - **相談ごと**:対応は任意。対応すると報酬(予算+/次フェーズの検収条件が易しくなる 等)
   - 差し込みは**約束外**なので断れる。ただし帰結(CS・負債・機会損失)は種類ごとに違う——拾うか見切るかの毎週の判断
   計画はここで崩れる(メモリ付き・可視・対処可能の原則は v3 から継続)
2. **朝会(同時配置)**:各自「今週の自分」を決める
   - **タスクに座る**(自分のスキルぶんキューブを積む)
   - **学習**(その系統のスキル+1。上限3)
   - **休憩**(疲労 -2)
   - **消火**(任意のタスクの🔥1個を除去)
   - 任意で**残業**(上記をもう1枠。即時疲労+1。疲労Lv2以上は禁止)
3. **週末**:キューブが必要数に達したタスクは納品判定
   (チームが「Lv1で納品/積み増し続行」を選ぶ)。納品したら:
   - 実行コスト(予算)を支払い、成果物をプロダクトボードへ(§2)
   - 参加者(その週までにキューブを積んだ全員)を記録

- 疲労:座った週ごとにタイルの疲労値(+1/重+2)。下手な人ほど週数がかかる=疲れる(実務どおり)
- 疲労Lv3 で限界イベント(従来どおり)。「品質の妥協」系は対象と影響を必ずログに出す

### 1-3. フェーズ終了

1. **約束の清算**:約束済み・未達の検収条件1つにつき CS-`commitPenaltyCs`(持ち越し。翌フェーズも未達なら再び)
2. **中間検収**:プロダクトボード(§2)をクライアントの重みで評価
3. 疲労の自然回復(-1)、次フェーズのスコープ会議へ
4. 最終フェーズ終了時は**最終検収**:プロダクトボード全体 × クライアント重みで CS を清算し、勝敗判定

## 2. プロダクトボード(出口。= お客様の成果)

- 作っている Web サイトの**完成図ボード**。スロット(サイトマップ/ワイヤー/デザイン/トップ/下層/CMS…)に、
  納品した成果物が **Lv付きで**はまっていく
- **検収**:フェーズ末、クライアントカードの重み(Q/C/D)が指す領域のスロットを見る。
  埋まっていない/Lv1 ばかりなら CS-、Lv2 が揃っていれば CS+
- **技術的負債の可視化**:Lv1 で埋めたスロットは炎上カードの対象になりやすい(雑に作った所は燃える)
- スロットには**完成ボーナス**を刷れる(例:「スタイルガイド完成:以後デザイン系の必要工数-1」)
  → 置き場所を選ぶ楽しさ(エンデバーの報酬レイヤー)はここに宿す。詳細は v4.1 でコンテンツ設計

## 3. メンバーと PM 帽子(職業クラス制の廃止)

「PM/デザイナー」という固定クラスをやめ、**能力(メンバーカード)**と**責任(PM 帽子)**を分離する。

### メンバーカード(能力=傾向を持った個人)

- 各プレイヤーに**メンバーカード**を1枚配る(配布 or ドラフト)。カードは
  **スキルプロファイル(勾配)+個人能力+経歴フレーバー**を持つ
- プロファイル例(合計3〜4、バリエーションで用意):

| 例 | 📋dir | 🎨des | ⚙eng | 個人能力(フェーズ1回。パッシブは常時) |
|---|---|---|---|---|
| 元エンジニアの何でも屋 | 1 | 1 | 1 | **マルチタスク**(パッシブ):残業の追加疲労なし |
| デザイン寄り | 1 | 2 | 0 | **磨き込み**:納品済みスロット1つに品質+1(Lv1→Lv2) |
| マネジメント寄り | 2 | 1 | 0 | **段取り**:今週、自分の積むキューブ+1 |
| 開発寄り | 0 | 1 | 2 | **自動化**:タスク1つの必要工数 -1 |

- 0 の系統には座れない(自然なロック)。1 あれば座れる=遅くても手伝える
- **5人プレイはカードを1枚増やすだけ**(第5クラスの発明が不要になる)
- 学習でプロファイルは育つ(上限3)。「自分のビルドを作る」楽しみはここに

### PM 帽子(責任=役割)

- **PM マーカー**を1人が被る(セットアップで決定。教育モードではフェーズごとに回す選択ルール)
- 帽子に付く権限:スコープ会議の最終決定 / **交渉**(フェーズ1回:検収条件の約束を1フェーズ延長、
  またはスコープ会議でカード2枚引き直し)/ 追加請求 / 大炎上のターゲット裁定
- スキルと責任が独立するので、「PM 寄りスキルの人が帽子を被るとは限らない」状況も生まれる(これも実務)

## 4. そのまま残るもの

- **CS 単一勝敗トラック**+クライアント QCD 重み(検収と追加請求に適用)
- **予算**:納品時コスト、PM の追加請求(CS と交換。フェーズ1回)、外注(予算+CS でキューブN個を即積み。要調整)
- **炎上システム**:デッキ・大炎上・延焼・持ち越し(脅威レイヤーとして不変)
- **疲労・限界イベント**(残業禁止の読み替え含む)
- **個人目標(働き方の個性)・マイルストーン**
- **イベントソーシング・GameConfig 外出し**などアーキテクチャ原則すべて

## 5. 削除台帳(v3 → v4)

| 削除 | 理由/行き先 |
|---|---|
| 専門席・人手席 | 工数キューブに統合 |
| やっつけ解決+CS債務 | 不要(下手=遅いだけ) |
| 過剰スペック割引 | 不要(上手=速いが既に報酬) |
| 収穫の選択・じっくり(ツォルキン式拘束) | 積み増し=品質に統合 |
| 品質閾値の独立判定 | プロダクトボード検収に統合 |
| 締切許容数(deadlineAllowance)と締切利息の併存 | 「約束した検収条件が未達 = CS-1/フェーズ」1本に統合 |
| カード個別の依存チェーン | レーン文法(前列に1枚)に置き換え |
| 秘匿要件の受動ドロー | 要件定義アクション(v4.1 で再導入検討:座って1週=要件カードを選べる/飛ばすと強制) |
| EP トラック | 削除済み(v3) |
| 固定4クラス(PM/ディレクター/デザイナー/エンジニア) | メンバーカード(能力)+ PM 帽子(責任)に分離 |
| 「タスク全部やる」前提 | 検収条件ベース(タスク=手段。やらない自由が構造化) |

## 6. GameConfig(暫定値)

```
firePerRound: 1

noOvertimeAtFatigueLv: 2
restRecovery: 2
phaseEndRecovery: 1
extraBillingBudget: 3
extraBillingCsCost: 1
extraBillingPerPhase: 1
// タスク必要工数のレンジ: 2〜5(コンテンツ側)

```

## 7. 未決事項(実装前に決める/シミュレーションで測る)

- ❓ 検収需要とベロシティの比(需要/供給 ≒ 1.3〜1.5 を初期仮説。ボットシミュで勝率40〜60%に調整)
- ❓ 差し込みの頻度・重さ(拾う/見切るが五分五分の判断になる帰結設計)
- ❓ 文脈コンボ(同じ人が同じタスクを続けると+1)を入れるか → 暫定:入れない(v4.0 は最小で出す)
- ❓ 外注の形:「キューブN個を即積み(予算4+CS1)」で工数通貨に統合できるか
- ❓ プロダクトボードのスロット構成・完成ボーナス・同一スロットの複数手段タスクの具体(v4.1 のコンテンツ設計)
- ❓ 中間検収と最終検収の重み配分
- ❓ メンバーカードのプロファイル種数と配布方式(配布/ドラフト)、PM 帽子のローテーション
- ❓ 追体験モード/トリアージモードのプリセット差分、連続キャンペーン(v4.2)
