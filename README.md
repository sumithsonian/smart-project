# スマートプロジェクト(デジタル版)

Web制作会社を舞台にした4人協力型ボードゲーム。現在 **ステージ1(ルールエンジン + ホットシートUI)**。

- ルールの単一の正: [docs/RULES.md](docs/RULES.md)
- アーキテクチャ方針: [CLAUDE.md](CLAUDE.md)

## 起動手順

```bash
pnpm install
pnpm dev        # http://localhost:3000 でホットシートUIが起動
pnpm test       # エンジンのテスト(Vitest)
pnpm typecheck  # 型チェック
pnpm lint       # ESLint
```

## 構成

| パス | 内容 |
|---|---|
| `packages/engine` | 純TSルールエンジン(UI/DB非依存・副作用なし・シード付き乱数) |
| `apps/web` | Nuxt 4 ホットシートUI(1画面で4人分を操作するデバッグUI) |

## ホットシートUIの使い方

1. 設定パネルで GameConfig・シナリオ・シードを調整して「ゲーム開始」
2. フェーズ開始イベントを解決 → プランニング(タスクをクリック → 各個人ボードの「配置」)
3. 全員「準備完了」→ PM がタスク処理順を宣言(「自動(依存順)」が便利)
4. 「次のタスクを解決」を連打しつつ、イベント・要件カードを処理
5. フェーズ終了判定 → 次フェーズへ。第4フェーズ終了で勝敗判定
- 「Undo」は最後のアクションを除いてリプレイ。「JSON エクスポート」でアクションログを保存(バランス分析用)

## 暫定実装(RULES.md に明記がなく、実装で仮決めした点)

プレイテスト前に要確認。確定したら RULES.md に反映する。

| # | 項目 | 暫定実装 |
|---|---|---|
| 1 | 休憩・追加請求・学習の効果タイミング | 配置時に即時適用(配置後の回収は不可) |
| 2 | フェーズ開始の補充量 | 手持ちに +tokensPerPhase(未使用分は持ち越し)。解決済みタスクのトークンはストックに回収され手元には戻らない |
| 3 | 補充ペナルティの判定タイミング | フェーズ開始イベント解決後の疲労が Lv2 以上なら -1(§2-1 の順序どおり) |
| 4 | 品質判定の集計範囲 | そのフェーズに獲得した Lv2 成果物のみ(累積ではない) |
| 5 | スキル条件の判定 | 参加者(トークンを置いた人)の誰か1人が満たせばよい |
| 6 | 成果物の帰属 | チーム共有。ただし参加者を記録し、個人目標(品質の鬼)の判定に使う |
| 7 | 協業フラグ | 2人以上のトークンが必須(ボーナス方式は未実装) |
| 8 | 学習タイル | 容量制限なし(learningTiles は枚数表示のみ)。学習途中のトークンは回収可・スキルアップ時に消費 |
| 9 | 要件カード | 一度選んだカードはタスクに付いたまま(未解決で持ち越しても再抽選しない) |
| 10 | タスクの「疲労制限」条件 | 未実装(タイル定義に該当フィールドがないため) |
| 11 | PM の処理順宣言 | 同列だけでなく全未解決タスクの順列を宣言(依存順はエンジンが検証) |

## 既知の制限(ステージ1のスコープ外)

- ネットワーク対戦・Supabase 連携なし(ステージ2)
- 5人プレイ未対応(エンジンは人数可変設計、コンテンツが4ロールのみ)
- カード・タイルはプレイテスト用の仮データ(バランス未調整)
- `redactFor` はエンジン API として実装・テスト済みだが、ホットシートUIでは未使用(全公開)
